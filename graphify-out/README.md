# `graphify-out/` — Knowledge Graph for `ai/docs/`

This directory contains a **navigable knowledge graph** built from the 117-document corpus under `ai/docs/`. The graph captures concepts, relationships, design rationale, and cross-document connections — including ones no human read and labelled by hand. It exists so that an LLM agent (or a human curator) can query the corpus structurally, find load-bearing abstractions, and trace surprising bridges across the harness-engineering, context-engineering, subagent, skills, hooks, agent-protocol, and SDD literatures.

The graph is the empirical companion to the synthesis written in `dev/research/2026-04-25-harness-engineering-research.md` — start there for the prose narrative; come here for the structural map.

> **Generated by**: [graphify](https://github.com/safishamsi/graphify) — a tool that turns any folder of files into a knowledge graph with community detection, an honest audit trail (every edge tagged EXTRACTED / INFERRED / AMBIGUOUS), and reproducible outputs.

## What this graph is for

Three things this graph does that reading `ai/docs/` linearly cannot:

1. **Persistent shared memory.** Once built, the relationships persist across sessions. Ask agents questions weeks later without re-reading 272 K words.
2. **Honest audit trail.** Every edge is tagged `EXTRACTED` (explicit in source), `INFERRED` (reasoned from context), or `AMBIGUOUS` (flagged for review). You always know what was found vs reasoned. The 89/10/1 split here means most relationships are grounded.
3. **Cross-document surprise.** Community detection surfaces connections between concepts in different files that nobody would ask about directly — e.g. *Spec-Driven Development* ↔ *CFG Constrained Decoding*, or *Phase-Based Context Loading* ↔ *Research-Plan-Implement Workflow*. These bridges drive the architectural insights captured in the research document.

Recommended use cases:

- **Onboarding an LLM agent** to harness-engineering work. Point it at `wiki/index.md` and let it crawl the community articles for the topics it needs.
- **Architecture conversations.** Use the god-node list and community labels to anchor "what are the load-bearing abstractions" discussions.
- **Doc audits.** Re-run graphify after editing `ai/docs/` and diff the god nodes / community structure — drift signals docs that have lost coherence.
- **Cross-source synthesis.** Surprising-connection edges are explicit pointers for "these two ideas should be reconciled in writing."
- **MCP-backed agent retrieval.** `graph.json` can be served via a graphify MCP server (`graphify --mcp`) to expose `query_graph`, `get_node`, `get_neighbors`, `get_community`, `god_nodes`, `graph_stats`, `shortest_path` to any MCP-compatible agent.

## How the wiki works

`wiki/` is an **agent-crawlable** knowledge base auto-generated from the graph:

- `wiki/index.md` is the entry point — it lists every community and god node with `[[wiki-link]]` references (Obsidian-compatible).
- Each community gets one article (`wiki/<Community Label>.md`) with the community's nodes, internal edges, and pointers to god nodes inside it.
- Each god node also gets its own article (`wiki/<Node Label>.md`) listing its connections.
- Articles cite `source_file` and `source_location` from the graph so the agent can re-read the original document for any specific claim.

To use it: open `wiki/index.md` in any agent harness that supports `[[wiki-link]]` resolution (Obsidian, Foam, or any LLM agent that can resolve relative paths). Agents that just read Markdown will still get useful structure from the human-readable headings.

## Suggested next steps for an LLM agent reading this graph

1. Read `dev/research/2026-04-25-harness-engineering-research.md` for the prose synthesis of what's in the corpus and how it maps onto a harness engineering reference guide.
2. Open `wiki/index.md` and follow the community link relevant to your task (e.g., a hook-related task → `wiki/Cursor_Hooks_Guide.md` plus `wiki/Agent_Skills_Standard.md` plus `wiki/Context_Engineering_Foundations.md`).
3. For each finding, cite back to the underlying file in `ai/docs/<theme>/<file>.md` using `path:line` notation. The wiki articles include those citations.
4. If the answer requires multi-hop reasoning, run `/graphify path "<concept A>" "<concept B>"` to find the shortest path between two concepts.
5. If your conclusion contradicts an INFERRED or AMBIGUOUS edge, surface that disagreement explicitly — those edges were flagged precisely because they need human or agent review.

## Reproducing or updating this graph

The graphify pipeline is invoked through the `/graphify` skill bundled at `~/.claude/skills/graphify/SKILL.md`. Common operations:

```bash
# Full rebuild from scratch (deep mode for richer INFERRED edges)
/graphify ./ai/docs --mode deep

# Incremental update — only re-extracts new/changed files
/graphify ./ai/docs --update

# Regenerate the wiki only (uses existing graph.json)
/graphify ./ai/docs --wiki

# Re-run clustering on existing graph (cheap, no LLM cost)
/graphify ./ai/docs --cluster-only

# Add a new external resource (URL, paper, tweet) and update
/graphify add https://example.com/paper

# Query the graph in BFS mode (default — broad context)
/graphify query "How does progressive disclosure relate to subagents?"

# Query in DFS mode (specific path tracing)
/graphify query "How does CLAUDE.md size limit propagate to skills?" --dfs

# Find shortest path between two concepts
/graphify path "Progressive Disclosure" "Lost in the Middle"

# Get a plain-language explanation of one concept
/graphify explain "RPI Workflow"
```

Costs (this run): **18.5 K input tokens, 4.2 K output tokens** for the full 117-file deep extraction. Incremental updates on small doc changes typically run for under 2 K total tokens because most files hit the per-file cache in `cache/`.

## Honesty notes

These caveats are inherent to any LLM-extracted graph and are surfaced explicitly so downstream consumers do not over-trust it:

- **Edge confidence is sampled, not measured.** Confidence scores reflect the extracting model's self-reported certainty. EXTRACTED edges (89%) have direct textual evidence; INFERRED edges (10%) are model judgments; AMBIGUOUS edges (1%) are flagged for review. Treat INFERRED/AMBIGUOUS edges as hypotheses, not facts.
- **180 nodes are isolated** (≤1 connection). Some are genuine fringe concepts; some are extraction artifacts. The wiki will not surface these — query `graph.json` directly if you need the long tail.
- **Three singleton communities** (IDs 18-20) — *JSON Schema Behavior*, *Refusal Edge Case*, *MCP Runtime* — are noise candidates flagged automatically. They have cohesion 1.0 because a single node cannot be incoherent, but that does not mean they are well-grouped.
- **The corpus snapshot is from 2026-04-25.** As `ai/docs/` evolves, run `--update` to keep the graph current. Do not assume the graph reflects the latest source state without checking `manifest.json` against `git log -- ai/docs/`.
- **One sensitive file was skipped** during detection (`ai/docs/context-engineering/million-token-context-window-syntackle.md`). It is not represented in the graph.
